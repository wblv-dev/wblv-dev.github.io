Domain Security
Toolkit

An automated domain security auditing tool that runs 35+ checks and produces customer-ready reports — built against real client domains where the findings were production issues, not synthetic test cases.

↗ GitHub Repository

The problem it solves

Domain security posture is one of the most consistently weak areas across SME clients. Email authentication is misconfigured, TLS is on outdated versions, HSTS isn't enforced, DNSSEC isn't deployed, and nobody has a clear picture of what's exposed. Running manual checks across 50+ domains per client is not sustainable.

This tool was built to solve that specific problem — not as an academic exercise, but because it was needed. The first version ran against wblv.uk and wellbelove.org, found real issues (DMARC gaps, TLS 1.0 still enabled, HSTS disabled, Cloudflare in Flexible SSL mode), and produced the remediation guidance that fixed them. It then ran against customer domains as part of a WAF baseline project covering approximately 176 Cloudflare-managed domains.

Origin: This wasn't built to be a portfolio piece — it was built because manually auditing domain security at scale is tedious and error-prone. The portfolio value is a side effect.

What it checks

The tool runs checks across six categories, with findings mapped to the relevant standard or framework where applicable.

TLS Configuration

  • Minimum TLS version enforcement (1.2 minimum, 1.3 preferred)
  • Cipher suite strength — detection of weak/deprecated ciphers
  • Certificate validity, expiry, and chain completeness
  • HSTS header presence and configuration (max-age, includeSubDomains, preload)
  • Cloudflare SSL/TLS mode detection — flags Flexible mode which breaks end-to-end encryption

Email Authentication

  • SPF record presence, syntax validation, and mechanism analysis
  • DKIM — checks for common selector presence and key strength
  • DMARC — policy level (none/quarantine/reject), pct value, reporting addresses
  • BIMI — brand indicator presence (optional check)
  • MTA-STS — mail transport security policy detection

HTTP Security Headers

  • Content-Security-Policy — presence and basic directive analysis
  • X-Frame-Options / frame-ancestors
  • X-Content-Type-Options
  • Referrer-Policy
  • Permissions-Policy

DNS Security

  • DNSSEC — validation chain from root to domain
  • CAA records — certificate authority authorisation
  • Dangling DNS — detection of DNS records pointing to unclaimed infrastructure
  • Zone transfer attempt (informational, not exploitative)

Certificate Intelligence

  • CT log enumeration via crt.sh — surfaces subdomains via certificate transparency
  • Certificate pinning detection
  • Multi-SAN certificate analysis

OSINT Exposure

  • WHOIS data exposure — checks for personal data in public WHOIS records
  • robots.txt — surfaces potentially sensitive path disclosures
  • Security.txt — RFC 9116 compliance check
  • Shodan integration (optional) — checks for known exposed services

Architecture

The tool is structured as a Python package with a modular checker architecture. Each check category is a separate module implementing a common interface — making it straightforward to add new checks without touching existing ones.

domain-security-toolkit/ ├── dst/ │ ├── __init__.py │ ├── cli.py ← Entry point, argument handling │ ├── runner.py ← Orchestrates check execution │ ├── report.py ← HTML report generation │ ├── checkers/ │ │ ├── tls.py ← TLS/certificate checks │ │ ├── email.py ← SPF/DKIM/DMARC checks │ │ ├── headers.py ← HTTP security header checks │ │ ├── dns.py ← DNS/DNSSEC checks │ │ ├── certs.py ← CT log enumeration │ │ └── osint.py ← OSINT/exposure checks │ └── templates/ │ └── report.html ← Jinja2 report template ├── tests/ └── pyproject.toml

Report generation

Output is a self-contained HTML report — no external dependencies, no CDN calls, no data sent anywhere. Each finding is classified as Pass, Warn, or Fail and mapped to a remediation recommendation aligned to the relevant framework.

Usage

# Install
pip install domain-security-toolkit

# Run against a domain
dst audit wellbelove.org

# Run with Cloudflare API integration
dst audit wellbelove.org --cloudflare-token $CF_TOKEN

# Output JSON for pipeline integration
dst audit wellbelove.org --output json

Real findings — wblv.uk and wellbelove.org

The first production run of the tool was against my own domains. The findings that came back were real issues that were subsequently fixed:

  • DMARC policy set to p=none — not enforcing, just monitoring. Upgraded to p=quarantine then p=reject.
  • TLS 1.0 still enabled — Cloudflare defaulting to broad compatibility. Minimum set to TLS 1.2.
  • HSTS not configured — missing header entirely. Added with 6-month max-age and includeSubDomains.
  • Cloudflare SSL mode: Flexible — traffic between Cloudflare and origin was unencrypted. Changed to Full (Strict).
  • No CAA records — no restriction on which CAs could issue certificates for the domain.
  • security.txt missing — RFC 9116 non-compliant. Added at /.well-known/security.txt.
Running the tool against your own infrastructure before running it against a client's is the only intellectually honest way to use it. You either fix what it finds or you understand why you've accepted the risk.

Cloudflare compliance reporting

A secondary use case emerged from the WAF baseline project — verifying that ~176 Cloudflare-managed customer domains were compliant with an agreed security baseline. The tool's Cloudflare API integration queries TLS configuration, managed rulesets, and custom WAF rule coverage per domain and produces a compliance report against the baseline standard.

The formal domain onboarding procedure written alongside this tool specified that every new domain added to the customer's Cloudflare account must pass the compliance report before being considered operationally onboarded.
